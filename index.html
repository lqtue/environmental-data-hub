Theo dõi Bão

Nguồn Chính: JTWC

Trung tâm Cảnh báo Bão Liên hợp (Hải quân/Không quân Hoa Kỳ) công bố các cảnh báo bão nhiệt đới Tây Thái Bình Dương ở ba định dạng:

Khám phá các cơn bão đang hoạt động:

https://www.metoc.navy.mil/jtwc/rss/jtwc.rss

Vị trí hiện tại + dự báo 5 ngày cho mỗi cơn bão:

https://www.metoc.navy.mil/jtwc/products/wp{NN}{YYYY}.txt

Bao gồm: vị trí, sức gió duy trì tối đa (knots), áp suất mực nước biển tối thiểu (mb), và dự báo tại TAU 12/24/36/48/72/96/120 giờ.

Định dạng Google Earth với bán kính gió và đường đi tốt nhất (best track):

https://www.metoc.navy.mil/jtwc/products/wp{NN}{YYYY}.kmz

Sử dụng cho phạm vi theo dõi một sự kiện cụ thể (đa giác bán kính gió, dải nguy hiểm).

Nên dùng định dạng nào: Tự động/theo lịch → cảnh báo văn bản (nhanh hơn, không cần giải nén). Đưa tin một sự kiện → KMZ (đa giác bán kính gió, dải nguy hiểm).

Dự phòng: JMA

Khi JTWC không khả dụng, JMA RSMC Tokyo cung cấp vị trí hiện hành (không dự báo):

https://www.jma.go.jp/bosai/typhoon/data/targetTc.js

Tệp này chỉ tồn tại khi có bão đang hoạt động.

Cách script hoạt động

  1. Giải nén KMZ — KMZ là file ZIP chứa một file KML bên trong. Script dùng zipfile để đọc file .kml.
  2. Phân tích KML — dùng xml.etree.ElementTree để duyệt tất cả các <Placemark> trong file KML.
  3. Phân loại Placemark — với mỗi placemark, script nhận diện loại dựa trên tên:
    • Tên chứa "DD/HHZ - NN knots" → forecast_point (dùng regex)
    • Tên chứa "RADIUS OF NN KT WINDS" → wind_radii
    • Tên khớp mẫu "YYMMDDHH Z" → best_track_point
    • Placemark chứa <LineString> → forecast_track hoặc best_track
    • Placemark chứa <Polygon> → danger_swath
  4. Chuyển đổi thời gian — tất cả thời gian UTC được chuyển sang GMT+7 và định dạng kiểu tiếng Việt (T2=Thứ 2, ..., CN=Chủ nhật). Ví dụ: "T3, 24/3 07:00".
  5. Chuyển đổi đơn vị — sức gió từ knots sang km/h (nhân hệ số 1.852).
  6. Phân loại cấp bão — theo thang NCHMF Việt Nam (xem bảng Phân loại Sức gió bên dưới).
  7. Ghi GeoJSON — tất cả features được gộp vào một FeatureCollection duy nhất kèm block _meta chứa thống kê.

Cách script hoạt động

  1. Tải dữ liệu — đọc shapefile IBTrACS (điểm), lọc theo danh sách SID trong LandfallTyphoon.csv.
  2. Chuyển đổi — thời gian UTC → GMT+7, sức gió knots → km/h, tọa độ → hệ chiếu mét (EPSG:32648).
  3. Xây dựng đoạn đường — nối mỗi cặp điểm liên tiếp thành LineString. Tính độ dài đoạn (m) và thời gian đoạn (h).
  4. Giao cắt hình học — dùng gpd.overlay() để tìm phần đoạn đường nằm trên đất liền Việt Nam. Tỷ lệ giao cắt xác định thời điểm đổ bộ chính xác (nội suy tuyến tính).
  5. Xác định tỉnh — dùng gpd.sjoin() với ranh giới tỉnh để tìm tỉnh đổ bộ đầu tiên và tất cả các tỉnh bão đi qua.
  6. Tổng hợp — với mỗi SID: thời điểm đổ bộ, vị trí (lat/lon), sức gió lúc đổ bộ, thời gian trên đất liền, sức gió trung bình, khoảng cách ngày giữa các lần đổ bộ trong cùng năm.

fetch_rain.py — Dị thường lượng mưa NASA POWER 
pip install geopandas pandas numpy requests scipy matplotlib shapely
python typhoon/fetch_rain.py 20260901 20261115
# Đầu ra: rain/RainAnomaly.geojson (mất khoảng 5-10 phút)

Nếu không truyền tham số ngày, mặc định là 1/1 – 30/11 năm hiện tại.

Yêu cầu đầu vào

Đặt tệp CountryBoundary.geojson (ranh giới quốc gia Việt Nam) ở thư mục gốc repo. Tệp này không có sẵn trong repo — có thể tải từ Natural Earth hoặc GADM.

Cách script hoạt động

  1. Tạo lưới điểm — phủ lưới 0.5° (~50 km) lên ranh giới Việt Nam → ~300 điểm.
  2. Tải song song — 10 worker song song, mỗi điểm gọi 2 API NASA POWER:
    • Lượng mưa thực tế trong khoảng thời gian chỉ định (API temporal/daily)
    • Trung bình 30 năm theo tháng (API temporal/climatology)
  3. Tính dị thường — anomaly = (thực tế − kỳ vọng) / kỳ vọng × 100%. Giá trị dương = ướt hơn bình thường.
  4. Nội suy — từ 0.5° sang 0.05° (~5 km) bằng nội suy bậc 3 (cubic), lấp khoảng trống bằng nearest.
  5. Tạo polygon — dùng matplotlib.contourf() để tạo các đường đẳng trị mỗi 10%, chuyển thành polygon, cắt theo ranh giới Việt Nam.

Tải dữ liệu IBTrACS

Hướng dẫn tải IBTrACS (NOAA)

IBTrACS là bộ dữ liệu đường đi bão toàn cầu, cập nhật hàng tuần trong mùa bão. Tải shapefile Tây Thái Bình Dương:

  1. Vào https://www.ncei.noaa.gov/data/international-best-track-archive-for-climate-stewardship-ibtracs/v04r01/access/shapefile/
  2. Tải IBTrACS.WP.list.v04r01.points.zip (điểm — dùng cho process_landfall.py và prepare_dashboard_data.py)
  3. Tải IBTrACS.WP.list.v04r01.lines.zip (đường — tùy chọn, để vẽ track nhanh)
  4. Đặt vào thư mục TyphoonRaw/ và đổi tên thành points.zip / lines.zip
# Hoặc tải bằng command line:
curl -o TyphoonRaw/points.zip \
  "https://www.ncei.noaa.gov/data/international-best-track-archive-for-climate-stewardship-ibtracs/v04r01/access/shapefile/IBTrACS.WP.list.v04r01.points.zip"

Trạm Đo Mưa (VNDMS)

Endpoint

GET https://vndms.gov.vn/warning_rain?lv1={min_mm}&lv2={max_mm}&types={1-7}

Trả về GeoJSON FeatureCollection các trạm đo mưa, lọc theo tổng lượng mưa và mạng lưới nguồn. Không cần key, không cần header đặc biệt.

Tham số

Tham sốKiểuPhạm viMô tả
lv1integer0 – 1000Ngưỡng mưa tối thiểu (mm) — chỉ trả về trạm có tổng mưa ≥ lv1
lv2integer0 – 1000Ngưỡng mưa tối đa (mm) — chỉ trả về trạm có tổng mưa ≤ lv2

Dùng lv1=0&lv2=1000 để lấy tất cả trạm bất kể lượng mưa.

Cấu trúc Feature

{
  "type": "Feature",
  "geometry": { "type": "Point", "coordinates": [106.598, 10.888] },
  "properties": {
    "label":      "Hóc Môn",       // Tên trạm
    "longtitude": 106.598,          // Lỗi chính tả từ API — đúng là "longitude"
    "latitude":   10.888,
    "popupInfo":  "<table>...</table>"  // HTML blob — cần parse
  }
}

Trường popupInfo là HTML thô chứa: tên trạm, mã trạm, địa điểm, tổng lượng mưa 24h / 3 ngày / 7 ngày, tên nguồn, và link biểu đồ lượng mưa theo giờ.

Ví dụ sử dụng Python

import requests
from bs4 import BeautifulSoup

def fetch_rain_stations(types: int, lv1: int = 0, lv2: int = 1000) -> list[dict]:
    url = "https://vndms.gov.vn/warning_rain"
    resp = requests.get(url, params={"lv1": lv1, "lv2": lv2, "types": types}, verify=False)
    features = resp.json().get("features", [])
    rows = []
    for f in features:
        p = f["properties"]
        soup = BeautifulSoup(p["popupInfo"], "html.parser")
        cells = [td.get_text(strip=True) for td in soup.find_all("td")]
        info = dict(zip(cells[::2], cells[1::2]))  # key-value từ các cặp td
        rows.append({
            "name":       p["label"],
            "lon":        p["longtitude"],
            "lat":        p["latitude"],
            "station_id": info.get("Mã trạm", ""),
            "location":   info.get("Địa điểm", ""),
            "rain_24h":   info.get("24 giờ", ""),
            "rain_3d":    info.get("3 ngày", ""),
            "rain_7d":    info.get("7 ngày", ""),
        })
    return rows

# Lấy tất cả trạm Vrain (types=3)
stations = fetch_rain_stations(types=3)
print(f"Số trạm: {len(stations)}")   # ~1377

# Lọc trạm có mưa > 50mm trong 24h
heavy_rain = fetch_rain_stations(types=3, lv1=50, lv2=1000)
Lưu ý
  • SSL tự ký (self-signed). Server dùng chứng chỉ không tin cậy. Cần verify=False trong requests (hoặc thêm CA vào bundle). Thêm import urllib3; urllib3.disable_warnings() để tắt cảnh báo.
  • Lỗi chính tả trong API: trường tọa độ là longtitude (thiếu chữ "i") — không phải lỗi code của bạn.
  • Không có trường thời gian (timestamp) trong response — dữ liệu phản ánh trạng thái hiện tại tại thời điểm gọi.
  • types=6 luôn trả về rỗng — bỏ qua.

Cách tính Độ Dị thường (Anomaly)

Đối với mỗi điểm lưới trên toàn Việt Nam ở độ phân giải 0.5° (~300 điểm):

  1. Tải lượng mưa hàng ngày trong khoảng thời gian diễn ra sự kiện
  2. Tải mức trung bình tháng trong 30 năm
  3. Dị thường = tổng thực tế − tổng kỳ vọng (số dương = ướt hơn bình thường)
  4. Nội suy từ độ phân giải 0.5° sang 0.05°, cắt xén theo ranh giới Việt Nam

Chuyển đổi Xe điện

Phương pháp LCA Việt Nam

Phân tích vòng đời (LCA) so sánh tổng phát thải GHG từ nôi đến mộ (cradle to grave) của xe điện (BEV), xe hybrid sạc điện (PHEV) và xe xăng/dầu (ICEV) trong điều kiện Việt Nam.

Điểm khác biệt của Việt Nam
  • Lưới điện nặng than — than chiếm 48% sản lượng điện (2024); hệ số phát thải ước tính 0,75–0,85 tCO₂e/MWh. Q1/2025 (mùa khô) than lên tới 56,5%. Đây là yếu tố quyết định việc xe điện có thực sự xanh không.
  • Thị trường chủ yếu là xe máy — 93,3% phương tiện đăng ký là xe máy. Mọi LCA có ý nghĩa phải bao gồm cả hai bánh, không chỉ ô tô.
  • VinFast là nhà sản xuất EV nội địa chủ đạo — 87.000 xe điện bán năm 2024 (+150% YoY), chiếm 17,6% thị phần ô tô; 400.000 xe máy điện năm 2025 (+532% YoY).

Dữ liệu Nước & Hồ chứa
  • File CSV phình to không giới hạn. water_data_full_combined.csv tích lũy các dữ liệu đọc hàng giờ mà không tự xóa bớt. Cần phải cắt bỏ dữ liệu cũ hơn 30 ngày ở mỗi mùa lũ.
  • Chỉ giám sát 16 trạm và 22 hồ — đây là một tập hợp được chọn lọc, không phải toàn bộ mạng lưới quốc gia.
  • Endpoint VNDMS rất dễ lỗi. Yêu cầu phải giả mạo header. Nếu trang web thay đổi cách xác thực, crawler sẽ không hoạt động.
  • Thủy Lợi Việt Nam sử dụng HTTP (không phải HTTPS) — tiềm ẩn rủi ro bảo mật.
  • Thao tác ghi file CSV không nguyên tử (non-atomic). Nếu script bị crash khi đang ghi giữa chừng, file CSV có thể bị hỏng.
Dữ liệu Lũ & Sạt lở
  • Cảnh báo NCHMF chỉ là bản chụp nhanh (snapshots). Mỗi lần fetch hàng giờ sẽ lưu lại trạng thái hiện tại. Nếu lỡ một lượt fetch = mất dữ liệu.
  • CentralVN.Landslide chỉ giới hạn ở Huế + Đà Nẵng. Nếu muốn mở rộng sẽ cần bộ dữ liệu ranh giới xã mới.
  • Không có tính năng xử lý việc sáp nhập/thay đổi mã xã. Nếu NCHMF cập nhật mã số sau các đợt sáp nhập hành chính, việc ánh xạ lên bản đồ sẽ bị lỗi.
  • Dữ liệu vùng ngập lụt (VegaCosmos) không có API. Phải yêu cầu một cách thủ công qua mỗi sự kiện.
Dữ liệu Lượng mưa
  • Độ phân giải NASA POWER rất thô (0.5° ≈ 50 km). Chỉ tốt để xem tổng quan khu vực, không phù hợp cho bản đồ tác động chi tiết.
  • Mất từ 5–10 phút cho mỗi lần chạy do phải gọi API tuần tự khoảng ~300 lần.

Đề xuất Nâng cấp: Backend Thống nhất

Hướng phát triển tiếp theo

Các công cụ hiện tại được xây dựng theo từng câu chuyện riêng lẻ — mỗi dự án một crawler, một file CSV. Một backend thống nhất có thể giải quyết sự phân mảnh này:

  • Một cơ sở dữ liệu duy nhất (PostgreSQL + PostGIS) — quản lý bão, mực nước, hồ chứa, cảnh báo, đường đi lịch sử, lượng mưa trong cùng một schema
  • Mạng lưới trạm đo đầy đủ — toàn bộ các trạm VNDMS hiện có, không chỉ giới hạn ở 16 trạm
  • REST API — truy vấn mọi dữ liệu theo vị trí, khoảng thời gian, hoặc ID cơn bão qua một lệnh gọi HTTP duy nhất
  • Các crawler tự động cho mọi nguồn dữ liệu chạy ngầm qua GitHub Actions cron
  • Chế độ cảnh báo — tự động tăng tần suất cào dữ liệu (crawl) khi một cơn bão đi vào vùng theo dõi của Việt Nam (95–120°E, 5–28°N)
  • Giải quyết vấn đề phình to CSV — sử dụng bảng chuỗi thời gian (time-series) đúng chuẩn với index và chính sách giữ liệu (data retention)

Các giao diện frontend hiện tại có thể kết nối với API như vậy với rất ít thay đổi — chỉ cần chuyển từ đọc file CSV sang gọi API.
